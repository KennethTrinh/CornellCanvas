You need to make sure you've gone through all branches you could possibly go through in your method, so lets say you have if(x==1){ do this} else{ } you need to test both when x == 1 and when it doesn't equal 1 so it depends on how you write your append method to know how many test cases you need. Think of edge cases too like, what if it's empty, what if it's not? etc... You should just test that it correctly appends the node to the end of the linked list so let's say we had [1,2] and you appended 3 it should be [1,2,3] but notice how the tail would vary and how the .next values etc would change (these would usually be accounted for in your toString method and the reverse). Think of [] and you append 2, same case. But these all depend on how you write your method so that's a general good rule of thumb. Also you can't store two different data types in one linked list that's why we have the generics next to the linked list, so once you set the array to only allow integers, you can't add strings etc.. Also appending nodes of certain length don't matter at all, let it be length 0, length 20 as long as it appends correctly that's all that matters. (I would still test length 0 and length>0 in my test cases to be fully covered) For tostring rev it should be tested while you test your append methods and the rest of your methods.
