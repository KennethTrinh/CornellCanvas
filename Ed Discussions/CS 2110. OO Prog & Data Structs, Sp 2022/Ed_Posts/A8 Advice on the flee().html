We introduce the methods that can be used in writing method flee(). Then, we give some hints and suggestions.

===================

Method flee()’s parameter state, of type FleeState, describes Pres Pollack's environment. Every time she makes a move, this object automatically changes to reflect her new location.

This object includes the following methods:

  • Node currentNode(): Return the Node corresponding to Pres Pollack's location.

  • Node exit(): Return the Node corresponding to the exit from the Cavern.

  • Collection<Node> allNodes(): Return a collection of all traversable nodes in the graph.

  • int stepsLeft(): Return the number of steps that are left for Pres Pollack to get to the exit.

  • void moveTo(Node n): Move Pres Pollack to node n and pick up all gold on that node. Also, reduce the steps left by the weight of the edge from Pres Pollack's current location to n. This fails if n is not adjacent to the current location.

  • grabGold(). Do NOT call this method! No need to do so. Gold on a node is automatically picked up when Pres Pollack movesTo() that node.

    • ===================

    1. There is no single way to write method flee() that works well on all caverns (maps/graphs).

    Some of you may find better ways than we did and get higher scores —more coins.

    So, get innovative, try various things, post your best scores on Ed Discussion note for scores.

    Here are some points to consider.

    2. When fleet() is called, following the shortest path to the exit is guaranteed to get Pres Pollack out within the prescribed number of steps.

    Therefore, your first order of business is to make that simple solution work: Find the shortest path and traverse it.

    Write a method that given a path starting on Pres Pollack's current position, has her traverse that path and end up on exit node. It can be useful in lots of situations.

    Remember, when Pres Pollack moves to a new node, gold on the node are picked up automatically. You don't have to be concerned with picking them up.

    3. TRY DIFFERENT APPROACHES . Write a new method for each new approach you take, and always have a solution that works to fall back on.

    Remember, it is better to have a solution that always gets out in the prescribed number of steps than one that sometimes fails.

    4. TIME LIMIT . There is an elapsed time limit of about 10-12 seconds on any map. We need that so that it doesn't take hours to grade your submission. We will run on our faster computers, so if yours is a bit slow and old, perhaps 12-14 seconds will be OK.

    5. HELPER METHODS . Write as many helper methods as you want. It is in your best interests to specify them carefully . The staff cannot help you if there are methods without specifications --how can they help when they can't tell what the methods are supposed to do? How do you remember what they are supposed to do? We have instructed the staff not to answer questions when methods are not properly specified.

    6. CHANGE SHORTEST PATH ALGORITHM . Right now, the shortest path algorithm gives you a path from the current position to any one of your choosing. But you may want to change it or make a copy and alter the copy. For example, one version could find the shortest path to all nodes and return the HashMap that gives all the information. The best solutions often do this.
