I believe it has to do with Integer wrapper class caching. A bit of googling brought me here and it appears that the Integer cache only contains numbers between -128 and 127: https://wiki.owasp.org/index.php/Java_gotchas#Immutable_Objects_.2F_Wrapper_Class_Caching I'm not too sure on this though, so if anyone has a more complete answer that would be cool.

Author: Caleb Chin (student)

For Strings in java, I think all string literals are cached (no numerical limit like for Integers), so if a String literal you are trying to use already exists in the String pool it will refer to the same object. However, I think when you use the constructor to make a new string (i.e. String s = new String("hello")), it makes a new object regardless of whether the argument passed into the constructor is a String literal that already exists in the String pool. I got most of this info from this site: https://www.baeldung.com/java-string-pool, but it's up to you whether or not you want to trust it. You can also try to run some the examples they have on the site in JShell to see if this is true.
