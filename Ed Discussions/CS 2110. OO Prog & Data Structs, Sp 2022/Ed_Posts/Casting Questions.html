1. I think the answer is summed up in this JavaHyperText pdf regarding "Hiding a field". Basically, Java treats overridden fields and methods differently when you call them at runtime. Whereas methods work bottom-up based on the dynamic type of the object from which the method is called, fields are accessed bottom-up based on the compile-time type of the object. That's why a.x == 0 ; because the static type of a is First , Java starts looking bottom-up from the from the First class. (Note that overriding fields is generally discouraged, so this shouldn't be a common occurrence you should be running into!) 2. Remember that casting between classes does not change anything about the object itself at runtime; it merely provides a different perspective to the compiler. Casting a to First does not change the fact that a is still of type Second at runtime, and so the dynamic dispatch of the num() method call means that it will still call the num() in Second . But, because field accesses are dependent on the compile-time type, casting will indeed make a difference there.
