For a hash function to be good, it should do a couple of things. As the problem says, it should return values distributed over the entire range of ints, even for inputs that look very similar. It should also give the same output given the same input. It should also try to avoid collisions.

option a is bad because it doesn't give the same output given the same input.

option b is bad because it doesn't return values distributed over the entire range of ints, even for inputs that look very similar. (similar length strings have similar hash values)

option c is bad because it doesn't return values distributed over the entire range of ints, even for inputs that look very similar. (strings that ends in the same letter have the same hash values, and the range of Unicode codepoints does not span the integers)

option d is bad because it doesn't return values distributed over the entire range of ints, even for inputs that look very similar. (output values only range from 0 - 42)

So from process of elimination alone, e must be the answer. As for what particularly makes e good, it's a property of XOR that xor-ing a uniformly random bitstring (the arbitrary constant) with another bitstring (the input int) is also uniformly random. You can see that XOR and multiplication is also deterministic, meaning we will get the same output from the same input. Multiplying by a large prime somewhat helps with the distributed over the entire range of ints.
