So that you can do some work on paper until the lab server issue is resolved: Here's what the first activity for lab 7 looks like:

Here is the specification for a function that could be useful for fixing typos:
def replace_first(word, target, rep): """Returns: a copy of string word with the FIRST instance of string target in word replaced by string rep. Example: replace_first('THanks', 'H', 'h') -> 'Thanks' Preconditions: target has length >=1 and appears in word. As mentioned, all the arguments have type str. """ Design a suite of test cases
The first step to implementing a function so that it meets its specification (a.k.a. "spec") is to design test cases for that function. That is, you want a set of inputs to that function that covers all the possible kinds of legal inputs, together with the resultant outputs according to the spec. This sort of activity can require creativity as you imagine all the kinds of situations that can crop up! And in the real world, there's nothing that tells you "how many" test cases are needed.Having these test cases in hand will make testing and debugging the function, once you start implementing it, much smoother!
Fill in the table below with non-redundant test cases for replace_first . You may use the one given in the spec as one of your test cases.
Due to some limitations of the underlying verifier, such that it can't always tell that two test cases are conceptually different, you may also, for this lab, need to make your test cases non-overlapping , meaning that each test case should not share any features in common with the others.
We strongly suggest you first write in just one test case and hit submit , to make sure you understand what format is expected by the system. (Don't fill in all the boxes at once just to have all that work go to waste from a formatting issue!)
