is checks whether they have the same id. == by default compares the ids; but for many classes (eg lists), the __eq__ function has been overwritten to do something else. And the behavior of in depends on the behavior of __eq__ (except I guess if you overwrite __contains__ , based on the professor's answer)

AAAAAAARGG! Just irretrievably deleted my whole answer!!!!!! Grr. Trying again. ......... I should have read this question more carefully when writing up the final exam study guide. Using "in" would work .... IF you had already correctly defined _ _ eq _ _ for Products to ignore differences in quantity but check for differences in name ... but, question 6(d) did ask students to do that! So, under the condition that an _ _ eq _ _ method was written as specified in question 6(d), using "in" would have worked. Here we are showing success: class Product: """ Simplified version with fewer attributes""" def __init__(self, n, quantity): self.name =n self.quantity = quantity def __eq__(self, other): """Suppose we only care that the name matches""" return isinstance(other, Product) and self.name == other.name tp1 = Product("Think Python", 30) goods = [tp1] tp_newshipment = Product("Think Python", 50) # note the quantity can be different print(tp_newshipment in goods) # this is true!!! But if we didn't have an _ _ eq _ _ , or had an _ _ eq _ _ that also checked that the quantities are the same (this would be weird: why would a new shipment have the same quantity as the amount we already have in stock), the desired behavior of in telling us whether we had a product in our inventory would fail, as shown here: class Product2: """ Simplified version""" def __init__(self, n, quantity): self.name =n self.quantity = quantity # No __eq__ here! tp1 = Product2("Think Python", 30) goods = [tp1] tp_newshipment = Product2("Think Python", 50) # note the quantity can be different print(tp_newshipment in goods) # this will print False
