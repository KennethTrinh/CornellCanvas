You know, when you asked your question, I looked and also panicked, "Where is is_female coming from?". So, happens to the best of us! (Or at least also to me.) First, by "4th" do you mean "1st", by any chance? Or did you mean "3rd". The stack is going from top to bottom, so the frame for song() is the first one that is created, and that one has an is_female parameter. So does the third one. But the 4th one, for function she() , doesn't have an is_female parameter (variable) in it. One explanation Line 25 takes the value True and hands it to song() as input ... ----> which song() stores in its parameter, which is named is_female . Line 21, when we get to it, hands the value of is_female --- which is True , from what we just said --- to live_long() as input ... ---> which live_long() stores in its parameter, which happens to also be named is_female . So that's why all the live_long() frames have the is_female parameter set to True . Another explanation What is happening when Python sees the code 1-25 is: It reads all the function definitions in lines 1-23, nods its head sagely and says, "I got it", but doesn't actually do anything but remember those definitions. The first time it executes any line is when it gets to line 25. The execution of the call song(True) sends the value True as input to the function song() , which stores the value True into its parameter, named is_female . So the is_female in the frame for song() has value True . Executing line 21 in song() involves a call that looks like <some function>(is_female) . Whatever "some function" is, Python first figures out the value of its argument. In this case, is_female (in the frame for song ) is True . So, whatever "some function" is, it takes the value True as input and stashes this value in its parameter variable. In this case, "some function" is live_long() and the parameter for live_long() is called is_female . That is_female is the one you highlighted in the second live_long() frame.
