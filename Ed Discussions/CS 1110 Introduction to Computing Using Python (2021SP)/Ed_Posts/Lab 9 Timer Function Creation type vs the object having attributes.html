Hi, this is a good question! It is can often be confusing for beginning students to distinguish between types vs objects of that type. I've struggled for a useful analogy for years now, and here's my current working example: Think of a "type" or "class" as an abstract concept or definition, like homo sapiens. (We're getting all fancy with the Latin here precisely to make the class seem abstract and divorced from normal life.) You and I are specific objects, or instantiations, of the type homo sapiens. You and I have attributes with values, like the color of our hair (mine is black/greying). But homo sapiens isn't a physical thing in our universe, and doesn't have a specific hair color. Rather, homo sapiens is a definition that implies that everything that is an object of type homo sapiens has hair. Analogously, the class Timer is a type ( tempus fugit, say). Every object that is a timer has an attribute hours, with some value. But the class Timer itself is just a definition that makes "hours" a required characteristic of objects of its type. ...... all of this is a long way of saying: you need/should be handling an object of class Timer, not trying to access attributes of the type/definition itself. Example: 
t = Timer(3,14)

print(t.hours)   # this works

print(Timer.hours) # this gives error: type object 'Timer' has no attribute 'hours'
