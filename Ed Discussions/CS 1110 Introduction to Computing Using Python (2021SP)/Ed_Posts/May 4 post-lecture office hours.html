So what's the diff? 1. how you write them (indented in a class or not, whether you have the self parameter) 2. how a user calls them. Example: len is a non-method function. Call via len(" abc ") strip() is a method. Call as " abc ".strip() x = 5 print(len(x)) # x doesn't have a length, no type-checking of the arugment Method only exists for the type it's defined for. "semantics" looks nice. If eat were a non-method function: call would be like eat(<animal>, <grass>) . If it were a method: it would be <animal>.eat(<grass>) , which is arguably more sensible --- one is the actor and one is the input.

Contrariwise, a function should probably not be a method if: (a) it doesn't have any special "association" with a particular object. Like, len() can work on both strings and lists, so it's hard to see why one would define it as a method of strings and a method of lists. (b) none of its arguments would have a special "actor" or "subject" status.

How to access attributes of an object that's a value in a dictionary? class Account: def __init__(self, d, p="temp"): self.amount = d # d is initial deposit. self.passwords = [p] all_accounts = {} # keys: names, values: accounts all_accounts["Prof. Lee"] = Account(1001) all_accounts["Prof. Fan"] = Account(50000000) # let's donate 500 dollars to charity from Prof. Lee all_accounts["Prof. Lee"].amount -= 500 print(all_accounts["Prof. Lee"].amount) # all_accounts["Prof. Lee"].self.amount # <--- no such attribute self # add a password to Prof. Fan's all_accounts["Prof. Fan"] # <--- that's an Account object all_accounts["Prof. Fan"].passwords # <--- list all_accounts["Prof. Fan"].passwords.append('12345') print(all_accounts["Prof. Fan"].passwords)
