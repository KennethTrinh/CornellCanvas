Hi, the short answer is: Your task here is to rewrite code so it reproduces the behavior of the previous version as far as possible. So have your code essentially do with such cases what it would have done before. In real life, if you knew who all the end users would be, you would ask them if this is really what they wanted. But if you don't, then you don't know whether people will want to use process_line() for purposes other than computing showdowns --- maybe they would in fact want a list of all who decided to take a gap year, so they could send them a follow-up survey or something --- then what else can you do but try not to break any user assumptions? And the longest answer is: I wrote process_line() early on in the A3 development process, before the showdowns-between-pairs-of-Colleges idea had fully solidified in my head, as a way of exploring the data myself. So it's written to be agnostic about what purposes College objects would be put to. And then when it came to the final version of the showdown idea, I decided to use the process_line() I already had rather than build a special-purpose one. (I expect the experience of using a tool that's available in a way that it wasn't completely explicitly designed for is not new for people, which perhaps explains the existence of phrases like "35+ Genius Uses for Binder Clips At Home." You have vaguely tempted me to have part of A6 be to use subclassing to handle these special kinds of "not-applied-to" "colleges" ... can you imagine how one might do it? (I should say that I don't think we'll end up revisiting A3 again in A6, but who knows right now....)
