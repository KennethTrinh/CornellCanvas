We are getting a weird hard fault. We are running test_t0. From what we can tell, we are getting the first process_select call that grabs the first ready RT process, which runs up until LEDGreen_On. At this point, the unready RT queue is set to NULL. Sometime before that call, the PIT1 service routine is called once and the current_time is at 1 ms. At LEDGreen_On, there is another PIT0 interrupt that calls process_select. This time, before anything happens in process_select, the unready RT queue is now set to 0x4b3 <do_process_select+10>. It somehow went from NULL to this value, and it obviously causes an error. See below: All other values seem to be as expected. It is just this one variable that is somehow getting updated, and it seems to be something in 3140.s that is causing it. Any ideas? I also updated my 3140.s file with the new changes. Update, we realized it is something to do with our PIT1 setup. When we comment out the initialization of PIT1 in process_start, the test passes.

We figured it out. I have no idea why this happens, but here you go: inside our pit1 service routine, we were calling a function that incremented the current_time. This function took a pointer to a realtime, not a pointer to a volatile realtime. So, when we called it, we casted the volatile current_time to a normal realtime, then called the function. For some reason, this causes a completely separate variable to be set to some random instruction. I am actually curious what is going on under the hood to cause this, so any C experts, please explain.
