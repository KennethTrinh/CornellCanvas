I explained this poorly in discussion session, so let me try again... For any parameter that we pass into a fucntion, the function actually makes a copy of that parameter in the stack. Here is an example in the MCUXpresso, I created a second addNode function that simply passes the pointer to the node (instead of pointer to pointer to node). You can see that in the variables monitor, the address of list_start is 0x20005d4 but the head (which is suppose to be the same pointer) is in 0x2000f8c (which I believe is on the stack of that function). This means that while we are pointing to the same address, there are two references. Now, normally this isn't an issue if our linked_list is guaranteed NOT empty. A quick aside, I was asked if there is a printf similar method of checking whether the addNodes are being done correctly. Not sure about printf but if you type the variable name in the "global variables" monitoring panel, you can verify the values as shown. Not sure if the window is there by default but you can navigate to Windows >> show view >> global variables And.. back to the regular scheduled program... The tricky part lies in we want to design the function such that it can address all edge cases (i.e. when the head ptr is null in the beginning). You can imagine that we've set this head pointer now to point to the new node but once we return it, we lose that reference! By using a ptr to a ptr to the node, when we dereference via *ptr_to_list_start, we are able then link the new_node onto the list node ptr that we instantiated in main. Hopefully this helps :)
