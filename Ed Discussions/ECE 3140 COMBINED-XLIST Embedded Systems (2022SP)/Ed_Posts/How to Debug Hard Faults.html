Title: How to Debug Hard Faults

Author: Aaron Wilhelm (admin)

Date: 2022-03-22T07:23:23.007785+11:00

Category: Labs

Vote Count: 2

We've had many students have hard faults when working on their labs. Here's how to debug them:

  • After the hard fault occurs, take a look at the stack trace (Window->Show View->Other...->Debug)

  • Find the last line of code that you wrote that called the function. Something is wrong with this line.

  • Then look at each variable for that line. Chances are at least one of them is a pointer that is NULL (aka 0x0). If the line is using the PIT, then chances are the PIT was not set up properly (which causes a hard fault). Review proper set up of the PIT from lab 2

  • For the pointer that is NULL, use CTRL+F to find each reference to that variable in your code. Double check that you never set it to zero, it doesn't go out of scope, and that you didn't already free that pointer (or that section of memory) beforehand

    • As this is a common problem that is time consuming to solve, if you are going to ask for help in trying to find a hard fault we would like to see the following :

  • A screenshot of the stack trace when the hard fault occurs

  • The function that contains the line of code that is causing the hard fault

  • A screenshot of all variables that are used at that line of code right before the hard fault is triggered. Recall that you can see local and global variables by opening their views if they're not already visible

    • This will help us a lot to answer your questions quicker (and you may very well find your problem doing this process anyways!) so we greatly appreciate it.
