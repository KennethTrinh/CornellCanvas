To be super pedantic, it is debatable whether constructors (the topic of Thursday's lecture) can be considered methods. According to the Java Language Specification (the authority on all Java pedantry), constructor declarations "look like method declarations," but unlike methods they are not members, are not inherited, and are not subject to hiding and overriding (all things we'll get to in the next few weeks). I would consider them separate (though Wikipedia does not).

But certainly it's fair to say that methods are a broader term, and that a method can be considered a "function" or a "procedure" based on its return type. The distinction is useful for a few reasons:

  • Functions can be used in expressions (while a procedure call concludes a statement)

  • Procedures must have "side effects" to be useful, while some functions ("pure functions") only compute a value; the latter have useful mathematical properties which are the topic of CS 3110

  • Historically, languages like FORTRAN and Pascal handled functions and procedures very differently (but in Java, they are basically unified, being declared and invoked with identical syntax)

    • That being said, when most computer programmers (including me) use the word "function," they mean to include procedures as well.

    The pedantic definition of a method is simply "a class member representing executable code that can be invoked". Some languages (C++) have functions that are not class members; there, methods are often called "member functions" to distinguish them from "free functions." However, Java doesn't have free functions, so for us, all functions are methods.
