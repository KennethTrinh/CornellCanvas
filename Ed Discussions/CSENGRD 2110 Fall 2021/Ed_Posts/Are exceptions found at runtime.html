Author: Curran Muhlberger (admin)

Vote Count: 2

There is a way in which exceptions interact with the type system at compile-time, though. Certain exceptions in Java (though not ArithmeticException ) are classified as "checked exceptions". If a function could possibly throw a checked exception, then it must declare that fact, and any code that calls it must either catch that exception type or else declare that it can throw it too. These "catch-or-throws" rules are enforced by the compiler, so at runtime, you should never be surprised by an unhandled checked exception. The flip side of this compile-time safety regarding checked exceptions is that you are forced to handle all possible errors, which (while important for production systems) can get tedious for quick scripts and lead to bad habits. Reading input and performing file I/O is notoriously annoying in Java because IOException is a checked exception. We'll learn more about exceptions in a future lecture (with associated reading in JHT), so you don't need to worry about these details at this point.
