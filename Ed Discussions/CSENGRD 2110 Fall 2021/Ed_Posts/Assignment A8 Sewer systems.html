What you cannot and cannot do in writing your program . Most but all of thess points are discussed in the handout. But please read carefully anyway to help you remember the points. 4A. Do NOT use static fields (unless they are constants, which never change). Our grading program could run several sewer systems simultaneously, using different threads, and having a static field that can change will be disastrous. 4B. Declare as many instance fields as you want. Write as many methods, static or non-static as you wish. Comment fields appropriately and specifying method carefully. The staff is instructed not to help with fields or methods that are not appropriately defined (specified). 4C. Import any classes and interfaces that you need from the Java API. Don't import other things that you may find on the net because our grading program may not know about them. 4D. We told you not to put all your code in the two main methods, seek and scram, for the following reason . For maximum flexibility, write a dfs walk (or code to scram) in a separate method. Then, call that method from seek (or scram). Why? First, you may need some initialization of fields or other things to record the actual dfs walk takes place. This is rarely done properly when trying to put everything in find, making it recursive. Second, you may be writing two dfs walks. The first is the basic, the second is an optimized one. If you don't get the optimized one working properly, just before submitting, fix seek to call the first one, which works. YOU SUBMISSION SHOULD ALWAYS SUCCEED. Always program in a way that provides yourself with maximum flexibility, especially in a program like this, where you may be trying several different strategies. 5.
