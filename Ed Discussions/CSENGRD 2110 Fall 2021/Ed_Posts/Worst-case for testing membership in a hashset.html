A load factor is # of entries / length of array. With a load factor of 1/2, we can say for example that there is one entry in a hashset of length 2. The element is in the first spot of the array and you are looking for that specific element. In this hashset example, it would take O(1) time since the element is at the beginning of the hashset. However, let us imagine that there is a hash-set that is length 10 with 5 elements. This would mean that the load factor is 1/2. In this case, the elements are al scattered about. There is a chain of 3 elements in the second slot of the hash set, 1 element in the 5th slot, and 1 element in the 6th slot. You will need to be looping until the 6th slot if you are specifically looking for that hashcode. That would be O(6) which is constant time. Unless you are trying to find the element that is located at the very end of the hash-set, I think it would usually take less time than O(n) which is why the expected time to test for a membership of a hashset that has a load factor of 1/2 is constant time. If we were dealing with a load factor of 2 for a hashset, for instance, time complexity would be drastically different. There would be an average of 2 elements per slot of the hashset and it would take a while to loop through it.

Maybe this could help you see this better? 1/2 load factor means there is an average of one element per two hash-set spaces. 2 load factor means that there is an average of 2 elements per hash-set space. In a hash-set with a load factor of 2, you will need to be looping through 2 elements per hash-set space, which means you will be looping through way more than O(n) in the worst case.
