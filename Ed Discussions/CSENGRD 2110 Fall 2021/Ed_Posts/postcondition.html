Preconditions and postconditions are part of a method's specifications. They should always be documented in some form, though it is often rather informal. You may choose to use assert statements to verify a pre- or post-condition if it is cheap to do so (this is recommended, but do not confuse the verification with the condition itself).

We will learn about searching and sorting algorithms in a few weeks; they are good for illustrating preconditions and postconditions. For example,

  • A precondition for a binary search might be that "the array to search is sorted in ascending order". If the caller provides an array that isn't sorted, then the result of the function will be nonsense, and the whole program is broken.

  • A postcondition for a sorting routine would be "the elements in the array are sorted in ascending order". When the routine returns to the caller, then the user may assume that the array they provided is now sorted. If it is not, then there is a bug in the routine, and the whole program is broken.

    • In both of these examples, the condition is expensive to verify (you'd have to look at every element in the array to check that it is sorted at the start of the search method and at the end of the sort method), so there should be no asserts in the code corresponding to these conditions. Instead, you would write JUnit tests to verify these procedures outside of a running program.
