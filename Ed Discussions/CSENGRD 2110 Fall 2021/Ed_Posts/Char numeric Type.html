Author: Curran Muhlberger (admin)

Vote Count: 0

A char will be implicitly converted to an int for the purpose of performing arithmetic operations on it (addition, subtraction, multiplication, division, remainder, negation). This is not the case for boolean s. A char is considered to be narrower than an int . However, a char is neither narrower than nor wider than a short - they have mutually incompatible numerical ranges. That is why char is in its own row in the wideness diagram. When a character is considered as a number, Java uses its Unicode codepoint . ASCII encoding represents a very small subset of Unicode (just the first 127 entries, out of nearly 150,000), but it's the subset used most often in English writing. There's nothing "magic" about these numbers per se - you can look them up is the Unicode specification; when we use the term "magic number", we are referring to a style of coding where numerical constants are littered throughout the code with no meaningful names or explanations. For example, c >= 97 && c <= 122 is an expression that makes use of "magic numbers" - where did 97 and 122 come from? Better would be to use variables with names to capture the information you got by reading the specification: littleA=97; littleZ=122; ... c >= littleA && c <= littleZ . But even better is to leverage languages and libraries that already know about all these messy details: c >= 'a' && c <= 'z' .
