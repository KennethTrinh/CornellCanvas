Many large programs are plagued by schedule and cost overruns related to software. This has been a problem since the '50s (see Books, The Mythical Man-Month ), and unfortunately professional software engineers and project managers still have a lot of trouble. Modern software tools and development practices have made things better, but it's not clear that they've kept pace with the growing size and complexity of software systems. Military contracts are a common place to see failures (extensive sub-contracting makes interfaces more rigid and places greater demands on communication and encapsulation). I don't have first-hand experience with either of those two programs, but much has been written about them, and you should be able to find some good studies with a library search. Launching JShell like this: jshell -R -ea enables asserts at runtime. Run jshell --help to see its own documentation. The -R flag means "pass the next flag to the "runtime system" (aka Java Virtual Machine) that JShell will be using. The -ea flag is a standard JVM flag that means "enable assertions". Getter and setter names have nothing to do with security ("security by obscurity" is a bad policy regardless). Their names should be obvious and stay what those methods do, but at a level of abstraction matching the class , not low-level field assignments. They should describe the behavior of the object that the user is trying to leverage. The prefixes "get" and "set" imply field access; simply leaving off "get" elevates that to "property query," which, while basically the same thing, isn't tied to a particular set of fields. Setters are often a bad idea outside of what's called the "Builder Pattern;" you usually want to be thinking of higher-level mutations and transformations instead, which will naturally yield different names (think addTo() for a fraction, rather than setNumeratorAndDenominator() ).
