To answer this, we need to be more precise about what it means for a cast to be "okay". A downcast is allowed at compile-time if it is conceivable for a class to be defined that would have partitions for both the "source" and the "destination" types. A downcast will be successful at runtime if the value actually does have a partition for the "destination" type. In a sense, what you say is correct for runtime behavior: a concrete object must have been constructed at some point with a partition for the destination type, but then presumably that information got lost as the result of an upcast somewhere. However, remember that upcasts are allowed to be implicit, so this loss of information might not be obvious. For example, storing an InterestAccount into an array of Account s is effectively an upcast.
